`@Output` وقتی لازم می‌شود که **کامپوننت فرزند بخواهد به والد خبر بدهد یک اتفاقی افتاده**، بدون اینکه خودش مستقیماً والد را تغییر دهد.

مثلاً:

```ts
// child
@Output() postCreated = new EventEmitter<Post>();

addPost() {
  this.postCreated.emit(newPost);
}
```

```html
<!-- parent -->
<app-new-post (postCreated)="posts.push($event)" />
```

### چرا وقتی بدون Output هم شد، پس لازم است؟

احتمالاً تو یکی از این کارها را کردی:

```ts
// مستقیم لیست را به child دادی
@Input() posts: Post[];
```

و داخل child نوشتی:

```ts
this.posts.push(newPost);
```

این کار از نظر فنی ممکن است، ولی از نظر طراحی Angular خوب نیست، چون:

کامپوننت فرزند دارد **دیتای والد را مستقیم تغییر می‌دهد**.

یعنی `NewPostComponent` دیگر فقط مسئول ساختن پست نیست؛ دارد لیست پست‌ها را هم مدیریت می‌کند. این باعث می‌شود کامپوننت کمتر قابل استفادهٔ مجدد، سخت‌تر برای تست، و وابسته‌تر به والد شود.

### تفاوت ذهنی اصلی

بدون `@Output`:

```txt
Child → دیتای Parent را تغییر می‌دهد
```

با `@Output`:

```txt
Child → فقط خبر می‌دهد
Parent → تصمیم می‌گیرد چه کند
```

این دومی تمیزتر است.

### آیا جایی هست که هیچ جایگزینی نداشته باشد؟

از نظر فنی، تقریباً همیشه جایگزین‌هایی هست:

* سرویس مشترک
* state management مثل NgRx
* signals/store
* template reference variable
* دوطرفه‌سازی با `[(...)]`
* دسترسی مستقیم با `ViewChild`

اما در معماری معمول Angular، برای ارتباط:

```txt
Child → Parent
```

استانداردترین و درست‌ترین راه `@Output` است.

پس بهتر است بگوییم:
`@Output` تنها راه مطلق نیست، ولی **راه درست و رسمی برای فرستادن event از فرزند به والد است**.

### مثال ساده

فرض کن این ساختار را داری:

```txt
PostsComponent
 ├── PostListComponent
 └── NewPostComponent
```

`NewPostComponent` نباید خودش بداند لیست پست‌ها کجاست یا چطور ذخیره می‌شود. فقط باید بگوید:

```txt
"یک پست جدید ساخته شد"
```

بعد `PostsComponent` تصمیم می‌گیرد:

```ts
onPostCreated(post: Post) {
  this.posts.push(post);
}
```

### جمع‌بندی

`@Output` برای وقتی است که:

```txt
فرزند می‌خواهد به والد خبر بدهد
```

نه اینکه دیتای والد را مستقیم دستکاری کند.

تو بدون `@Output` هم توانستی انجامش بدهی، چون JavaScript اجازهٔ تغییر reference را می‌دهد؛ اما راه تمیز Angular این است که child فقط event بدهد و parent state را مدیریت کند.
